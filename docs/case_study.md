# Case Study：把「逐字稿翻找」變成可稽核的秒級問答

一頁版的需求 → 方案 → 成效 → 踩坑紀錄。說明這套系統是為了解決什麼問題、為什麼這樣設計、以及量化結果。

---

## 1. 情境與痛點

**目標使用情境**：投資研究 / 法遵單位的分析師，日常要從**幾十頁的法說會逐字稿、主管機關規章 PDF** 裡，找出某個數字或某段原文（例：「鴻海 2022 第四季毛利率多少？」「興櫃轉上市櫃幾家？」）。

**痛點**：
- 一份逐字稿捲動翻找一個數字要 **5–10 分鐘**；找到還要**人工核對頁碼出處**才能引用。
- 不同人找法不一致、容易看漏。
- 「直接用關鍵字搜尋」不夠：搜尋找不到語意（問「毛利率」漏掉寫「獲利率」的段落），而且要的是**「答案＋可稽核出處」**，不是一堆命中段落自己再讀。

**要解決的核心**：自然語言問 → **秒級得到答案＋頁碼出處**；文件沒有的要**明確說「查無」而非編造**。

## 2. 目標與成功標準

| 維度 | 驗收方式 |
|---|---|
| 找得到 | 檢索 **context recall**（答案有沒有被撈進來） |
| 答得對 | **answer correctness**（答案含正解） |
| 可稽核 | **citation 正確率**（頁碼落在正解所在頁） |
| 不亂編 | **faithfulness**（答案有依據）、**拒答正確率**（沒有的要拒答） |

**約束**：資料敏感（財務/法遵）→ **優先本地、不出機器**；零預算 → **CPU、開源模型**。

## 3. 方案決策（為什麼這樣選）

| 決策 | 為什麼 |
|---|---|
| **RAG 而非 fine-tune** | 知識會變（隨時加新文件）、要可引用出處、零標註成本 → RAG 勝 |
| **本地 CPU（Ollama + fastembed）** | 財務/法遵資料隱私 + 零預算 → 不送雲端 |
| **Hybrid 檢索（向量 + BM25）** | 純向量對精確詞/數字/股號 silent fail（如「2317」「979 億」）→ 加字面比對 |
| **cross-encoder 重排** | 小語料品質最大槓桿 |
| **引用護欄（剝除非命中頁碼）** | LLM 會產生「格式對但來源錯」的頁碼（citation-shaped 幻覺）→ 後處理擋掉 |
| **程式碼編排單步 Agent** | 小模型自由多步不可靠 → 流程寫在程式，每步只讓模型做一個決定 |
| **78 chunks 用 FAISS flat、不上向量 DB** | 此規模暴力精確搜即最佳；過早上 Qdrant 是 over-engineering（能說出何時該升級） |

## 4. 量化成效（golden set 28 題 + 檢索 n=12，皆附 bootstrap CI）

- **找得到**：context recall **1.000** — 答案幾乎都被撈進 context，**檢索不是瓶頸**。
- **答得對**：answer correctness **0.893** [0.786, 1.000]（factual/qualitative 類 100%）。
- **找得準**：hybrid 在檢索層 **ΔMRR +0.229（CI 不含 0，顯著）**。
- **可稽核**：給頁碼時引用正確率高；引用護欄會剝除範圍外頁碼。
- **體驗**：一題 30–60 秒（CPU，半互動）；頁碼出處可一鍵核對。

> 把分析師「翻找 + 核對出處」的 5–10 分鐘，壓到一次秒級問答 + 可點選的頁碼。

## 5. 踩坑 → 根因 → 修復（真實工程紀錄）

1. **頁碼造假**：早期取錯 metadata key 致頁碼 `-1`，LLM 仍吐似是而非頁碼 → 改用 `metadata['page_number']` + 加 runtime 引用護欄（比對被引頁 ∈ 檢索命中頁）。
2. **小樣本下 reranker 結論翻轉**：索引由 85→78 chunks 重建後，兩個 reranker 的 A/B 排序翻面 → 結論：n=12 不足以下終局裁定，需擴測試集；不對外宣稱「顯著」。
3. **新技巧效益被遮蔽**：hybrid 在檢索層顯著、端到端卻打平 → 拆成「檢索層 / 端到端」兩層級量測才看清，並理解其效益**隨語料規模上升**。
4. **用客觀指標反證主觀評審**：以本地小模型當 faithfulness judge 時，分數（0.667）明顯低於客觀正確率（0.893），抽查發現它把「答對且有依據」的也判不忠實 → **小模型 judge 不可靠**，數字僅供相對參考。

## 6. 限制與下一步（誠實）

- 語料**刻意保持 toy**（78 chunks）以驗證鏈路；測試集 n 小、CI 寬。
- 兩個可定位的生成端缺口：未來年度問題（2025 營收）拒答紀律、與少數數字抽取（資本支出 979 已在 context 卻沒抽出）→ 強化抽取/拒答 prompt。
- 規劃中：本地 vs 雲端模型路由（選型對照）、增量 ingestion、metadata 過濾；大語料/多租戶才需要的權限與服務化層為「已知設計、未實作」。

## 7. 可遷移的判斷

- **每個決策都量化**，不是「感覺」選的。
- **知道何時不該做什麼**（不過早上向量 DB、不放任小模型自由多步、不盲信 LLM judge）。
- **誠實標註不確定性**（小樣本、judge bias）——比假裝一切完美更可信。
